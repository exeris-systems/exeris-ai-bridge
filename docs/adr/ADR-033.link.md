# ADR-033 — `KernelDiagnostics` SPI — Read-Only Runtime Introspection for Agent and CLI Adapters (link stub)

**Authoritative copy:** [`exeris-kernel/docs/adr/ADR-033-kernel-diagnostics-spi.md`](../../../exeris-kernel/docs/adr/ADR-033-kernel-diagnostics-spi.md)

ADR-033 is a **cross-repo** decision owned by `exeris-kernel`. It defines the public, read-only SPI surface (`eu.exeris.kernel.spi.diagnostics.KernelDiagnostics`) that this repository consumes through its `kernel:*` MCP tool family.

## Why this stub exists

`exeris-ai-bridge` 0.4.0 was explicitly blocked on a `KernelDiagnostics` SPI per `ROADMAP.md` §0.4.0 and ADR-025 §Engineering Protocol item 2. ADR-033 is the contract that unblocks it: the four bridge tools (`kernel:list_providers`, `kernel:list_capabilities`, `kernel:get_bootstrap_dag`, `kernel:describe_subsystem`) map 1:1 onto the four SPI methods.

## When to consult ADR-033

- Implementing or modifying any tool handler under `src/tools/kernel/**`.
- Building `src/transport/kernel-adapter.ts` — the Node-side wrapper that spawns the kernel CLI and frames JSON over stdio.
- Any change to the contract assumed by bridge integration tests against the kernel CLI.
- Wall-integrity reviews — ADR-033 enforces process-boundary separation by construction, which is what preserves The Wall from this repo's side too (see [`CLAUDE.md` §"The Wall by construction"](../../CLAUDE.md)).

## Consumer-side obligations from ADR-033

- The Node adapter MUST treat the CLI as a long-lived child process with framed JSON-over-stdio; no shared classpath, no in-process kernel link.
- The adapter MUST honour the `schemaVersion` field on every snapshot response and reject responses with an unknown major version.
- Tool handlers MUST NOT attempt to bypass the CLI and reach into a co-located kernel directly — there is no co-located kernel from the bridge's perspective.
- Authenticated / remote transport is out of scope for ADR-033 and will arrive via a separate bridge-side decision (likely bridge 0.6+ transport-auth work).
